Como funciona el simulador

CloudSense genera metricas sinteticas que imitan el comportamiento de una instancia EC2 real. El modo de carga determina los rangos de CPU, RAM y red que se usaran en las proximas lecturas.

Instancia simulada: t3.medium en us-east-1. Se uso este tipo porque es el mas comun en despliegues AWS de proposito general: 2 vCPU, 4 GB RAM, precio moderado. En un entorno real, la plataforma se conectaria a CloudWatch para obtener metricas de cualquier instancia del cliente; aqui se simula ese flujo con datos sinteticos bajo los mismos rangos.

Modo CPU RAM Red entrada Red salida Descripcion
Normal 30 - 65 % 35 - 60 % Variable segun hora del dia Operacion rutinaria. Trafico de red proporcional a la hora (8am-5pm mas alto).
Alta carga 70 - 88 % 65 - 82 % 4 - 15 MB/s 6 - 25 MB/s Pico de demanda. Mas salida que entrada: el servidor esta respondiendo muchas peticiones.
Critico 88 - 99 % 82 - 97 % 10 - 35 MB/s 1 - 8 MB/s Saturacion. Mucha entrada (flood/DDoS), poca salida: el servidor no puede responder.
Nocturno 8 - 22 % 18 - 32 % 20 - 350 KB/s 8 - 120 KB/s Bajo uso. Solo healthchecks, cron jobs y tareas de mantenimiento.

Por que el historial no cambia al cambiar modo: cada lectura se escribe en la base de datos de forma permanente. El modo solo afecta la siguiente lectura que genere el scheduler (cada 5 minutos) o la que tu mismo fuerce con Forzar lectura. Este comportamiento replica sistemas de monitoreo reales como CloudWatch o Grafana: el historial es inmutable y auditables.

Para ver el efecto de un cambio de modo de forma inmediata, cambia el modo y luego presiona Forzar lectura. El nuevo punto aparecera al final de la grafica en el monitor.
